### Как общаются программы
​В эпоху зарождения интернета взаимодействие между компьютерами было устроено относительно просто. Однако с развитием веб-технологий и переходом от монолитных серверных приложений к распределенным системам возникла необходимость в стандартизации этого общения. Программам на разных компьютерах, написанным на разных языках программирования, требовался общий «язык» для обмена данными. Так появился API.

​С того момента стандартом в индустрии являлась архитектура REST (Representational State Transfer), в которой данные чаще всего передаются в текстовом формате JSON по протоколу HTTP/1.1. Этот подход завоевал мир благодаря своей простоте и читаемости: любой разработчик может открыть браузер, сделать запрос к серверу и глазами увидеть понятный текст:
```
{
	"user_id": 1477,
	"user_name": "kazicen52",
	"role": "user"
}
```

Однако с ростом масштабов интернет-гигантов (таких как Google, Netflix, Яндекс) и появлением сотен тысяч связанных между собой серверов традиционного REST + JSON стало не хватать. 

### Проблема традиционного REST API и формата JSON


#### Почему же проверенный годами инструмент перестал устраивать индустрию высоконагруженных систем? На это есть три фундаментальные причины: 
1. Избыточный объем текста. Текстовый формат JSON крайне неэффективен с точки зрения расхода сетевого трафика. Пересылая структуру данных, мы каждый раз вынуждены передавать текстовые названия полей ("user_id", "user_name"). Если между серверами компании ежесекундно летают миллиарды таких запросов, сеть перестаёт справляться.
2. Высокая нагрузка на процессор. Компьютеры не воспринимает текст, они работают с бинарным кодом. Чтобы отправить JSON, сервер должен перевести свои внутренние данные в текст. Принимающий сервер должен сделать обратную операцию - прочитать текст и распарсить его обратно в двоичный вид. На эту конвертацию процессоры тратят огромные вычислительные ресурсы.
3. Ограничения протокола HTTP/1.1. Этот протокол работает по принципу «один запрос - один ответ в один момент времени». Чтобы отправить несколько запросов параллельно, клиенту приходится открывать множество TCP-соединений, что нагружает операционную систему.


​Решение проблемы: В 2015 году компания Google представила открытую технологию gRPC (Google Remote Procedure Call), которая кардинально изменила подход к проектированию быстрых и эффективных распределенных систем. 

### Архитектура gRPC:

​Высокая скорость gRPC достигается за счет синергии двух технологий: формата Protocol Buffers и протокола HTTP/2. Разберем их внутреннее устройство.  
### ​1. Формат Protocol Buffers (Protobuf)  
​Вместо текста gRPC использует бинарный формат сериализации Protobuf. Все данные строго описываются в файле-контракте с расширением .proto:
```
message UserRequest {
	int32 user_id = 1;
	string user_name = 2;
}
```

Утилита-компилятор автоматически генерирует на основе этого файла готовый код для нужного языка программирования (Python, Go, Java). Механизм бинарного сжатия Protobuf держится на трех механизмах: 
* ​Отказ от имен полей: Названия полей (user_id) вообще не передаются по сети. Вместо них используются короткие числовые теги (= 1, = 2). 
* Структура TLV (Tag-Length-Value): Данные упаковываются в плотный поток байт, где для каждого элемента указывается только его цифровой тег + тип данных (Tag), длина для динамических типов вроде строк (Length) и само значение (Value).
* Умное кодирование чисел (Varints): Обычно тип int32 всегда занимает в памяти 4 байта. Алгоритм Varints в Protobuf ужимает маленькие числа (от 0 до 127) всего в 1 байт. Чем меньше число, тем меньше места оно занимает на диске и в сети.  
​Итог: Пакет данных сжимается в 3–5 раз по сравнению с JSON и обрабатывается процессором мгновенно, без парсинга текста.

### ​2. Протокол HTTP/2  
​gRPC работает строго поверх HTTP/2, который кардинально оптимизирует сетевой транспорт:
* ​Мультиплексирование: В рамках всего одного открытого TCP-соединения клиент и сервер могут одновременно отправлять тысячи запросов и ответов. Они нарезаются на мелкие кусочки и идут в перемешку, не блокируя друг друга.
*  Бинарные фреймы вместо текста: В отличие от HTTP/1.1, где команды передаются строками, в HTTP/2 всё общение идет в строго бинарных кадрах: фрейм HEADERS (служебная информация) и фрейм DATA (полезная нагрузка).
* Сжатие заголовков алгоритмом HPACK: Клиент и сервер создают общую таблицу заголовков. Если метаданные не меняются от запроса к запросу, они не пересылаются заново, передается лишь их цифровой индекс из таблицы.
* Потоковая передача: HTTP/2 позволяет gRPC поддерживать двусторонний стриминг. Сервер может открыть канал и непрерывно передовать данные клиенту без постоянных повторных запросов.


### Область применения gRPC

​gRPC не создавался для замены REST API во всем интернете. Браузеры плохо приспособлены для работы с бинарными потоками на низком уровне, поэтому на фронтенде REST API и JSON остаются лидерами благодаря простоте отладки.

​Основное задача gRPC - это внутреннее взаимодействие в микросервисной архитектуре. Когда одно крупное приложение разбивают на сотни мелких сервисов между ними циркулирует колоссальный объем трафика. Благодаря gRPC сервера общатются на максимальной скорости и экономя компании миллионы долларов на оборудовании.
### ​Преимущества и недостатки технологии
##### Ключевые плюсы:
* Производительность: Минимальные задержки (latency) и колоссальная экономия трафика.
* Кроссплатформенность: Один .proto файл позволяет автоматически сгенерировать строго типизированный код для серверов на Go, Python, Java или C++.
* Безопасность: За счет HTTP/2 технология по умолчанию ориентирована на шифрование TLS.
##### Ключевые минусы:
* Сложность отладки: Бинарный код нельзя прочитать глазами в консоли, для тестирования нужны специализированные утилиты (например, Postman новых версий или Evans).
* Порог входа: Требуется изучать синтаксис Protobuf и жестко следовать обратной совместимости контрактов.
### Итого 

​Технология gRPC представляет собой скачок в области архитектуры микросервисов. В современной ИТ-индустрии дела обстоят так: REST API остается главным интерфейсом для клиентских приложений, в то время как gRPC занял позиции внутри высоконагруженных микросервисных систем, обеспечивая молниеносное бинарное взаимодействие между серверами крупнейших ИТ-компаний.
